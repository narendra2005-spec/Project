# Software Requirements Specification (SRS)
## Project: In-Class Student Companion App

**Version:** 1.0  
**Date:** March 8, 2026

---

## 1. Introduction

### 1.1 Purpose
The purpose of this document is to define the software requirements for the **In-Class Student Companion App**. This application is designed to streamline the academic experience for students by providing a centralized platform for attendance tracking, course materials, deadlines, and performance monitoring.

### 1.2 Scope
The **In-Class** app is a cross-platform mobile application (built with React and Capacitor) that allows students to:
- Mark attendance in real-time using unique session codes.
- Access class notes and announcements from faculty.
- Track assignment deadlines and exam schedules.
- Monitor academic performance through internal assessment marks.
- Download and view course materials directly on their device.

### 1.3 Definitions, Acronyms, and Abbreviations
- **SRS:** Software Requirements Specification
- **App:** The In-Class mobile application
- **User:** A student enrolled in the institution
- **Faculty:** Academic staff who manage courses and sessions
- **Session Code:** A 6-digit unique code generated by faculty for attendance marking
- **Firebase:** The backend platform used for database and authentication

## 2. Overall Description

### 2.1 Product Perspective
The **In-Class** app functions as a client-side interface for students, interacting with a Firebase backend. It is designed to be part of a larger academic management ecosystem where faculty have a separate interface (web or app) to generate codes, upload notes, and input marks.

### 2.2 User Classes and Characteristics
- **Student:** The primary user. Tech-savvy, requires quick access to information, uses mobile devices primarily. Needs to mark attendance quickly during class.
- **Faculty (Backend/Admin):** Generates content consumed by the student app.

### 2.3 Operating Environment
- **Client:** Android and iOS devices (via Capacitor), Modern Web Browsers (Chrome, Safari, Edge).
- **Server:** Firebase (Firestore, Authentication, Hosting).
- **Network:** Requires active internet connection for real-time features.

### 2.4 Design and Implementation Constraints
- **Platform:** Must be built using React and Capacitor for cross-platform compatibility.
- **Database:** Must use Firestore for real-time data synchronization.
- **Offline Capability:** Critical features (like viewing downloaded notes) should work offline, though attendance marking requires a connection.

### 2.5 Assumptions and Dependencies
- **AS-01:** Users have access to a smartphone running Android 8.0+ or iOS 13.0+.
- **AS-02:** The institution provides a stable Wi-Fi network or students have mobile data access in classrooms.
- **AS-03:** Faculty members are responsible for the accuracy of session codes and marks entered into the system.
- **DE-01:** The application depends on the availability of Firebase Cloud Services.

## 3. Functional Requirements

### 3.1 Authentication
- **FR-01:** The system shall allow students to log in using their institutional email credentials.
- **FR-02:** The system shall maintain a persistent login state until the user explicitly logs out.

### 3.2 Dashboard
- **FR-03:** The system shall provide a main dashboard with quick access to four primary modules: Attendance, Notes, Deadlines, and Marks.
- **FR-04:** The dashboard shall display the student's profile summary (Name, Role, ID).

### 3.3 Attendance Module
- **FR-05:** The system shall allow students to enter a 6-digit code to mark attendance for a specific session.
- **FR-06:** The system shall validate the code against active sessions in the database.
- **FR-07:** The system shall prevent duplicate attendance marking for the same session.
- **FR-08:** The system shall display a calendar view of attendance history.
- **FR-09:** The system shall allow users to click on a date in the calendar to view detailed attendance records (Present/Absent status, Time marked).
- **FR-10:** The system shall display a list of currently active class sessions.

### 3.4 Notes (Class Stream) Module
- **FR-11:** The system shall display a chronological stream of notes and announcements from faculty.
- **FR-12:** The system shall allow filtering of notes by specific subject/group.
- **FR-13:** The system shall provide a search bar to find notes by title or content.
- **FR-14:** The system shall support file attachments (PDF, Images, Word, PowerPoint).
- **FR-15:** The system shall allow users to download attachments to their local device storage.
- **FR-16:** The system shall trigger the device's native file viewer to open downloaded documents.

### 3.5 Deadlines Module
- **FR-17:** The system shall display a list of upcoming assignments and exams.
- **FR-18:** The system shall visually distinguish between different types of deadlines (e.g., Assignment vs. Exam).
- **FR-19:** The system shall indicate overdue items with a distinct visual style (e.g., Red color).
- **FR-20:** The system shall sort deadlines by due date (nearest first).

### 3.6 Marks Module
- **FR-21:** The system shall display a scorecard for each subject.
- **FR-22:** The scorecard shall show marks for individual assessments (Test 1, Test 2, Test 3).
- **FR-23:** The system shall automatically calculate and display the final average based on the best two scores (or configured logic).

## 4. Non-Functional Requirements

### 4.1 Performance
- **NFR-01:** The app should load the dashboard in under 2 seconds on a standard 4G network.
- **NFR-02:** Attendance submission should be processed in under 1 second.

### 4.2 Reliability
- **NFR-03:** The system should have 99.9% uptime during academic hours (8 AM - 5 PM).
- **NFR-04:** Data synchronization should ensure that attendance records are never lost.

### 4.3 Security
- **NFR-05:** All data transmission must be encrypted using HTTPS/TLS.
- **NFR-06:** Student data (marks, attendance) must be accessible only to the authenticated student and authorized faculty.
- **NFR-07:** Session codes should expire after a set duration to prevent proxy attendance.

### 4.4 Usability
- **NFR-08:** The UI should be intuitive, requiring no training for students to use.
- **NFR-09:** The interface must be responsive and adapt to various screen sizes (Mobile, Tablet).

### 4.5 Scalability
- **NFR-10:** The system shall support up to 5,000 concurrent users marking attendance simultaneously.
- **NFR-11:** The backend architecture shall be capable of handling a 200% increase in data volume annually.

### 4.6 Maintainability
- **NFR-12:** The application shall use a modular component-based architecture for easy feature expansion.
- **NFR-13:** All critical business logic shall be documented within the code using standard JSDoc/TSDoc formats.

## 5. System Interface

### 5.1 User Interface
- **Style:** Clean, modern, "Student-First" design using Tailwind CSS.
- **Navigation:** Tab-based navigation for easy switching between modules.
- **Feedback:** Visual feedback (toasts, spinners) for all user actions (loading, success, error).

### 5.2 Software Interfaces
- **Firebase Firestore:** NoSQL database for storing user data, groups, notes, and records.
- **Capacitor Plugins:**
    - `@capacitor/filesystem`: For saving downloaded files.
    - `@capacitor/browser`: For opening external links.
    - `@capacitor-community/file-opener`: For opening documents in native apps.

### 5.3 Communication Interfaces
- **CI-01:** The application shall communicate with Firebase Firestore and Auth via the Firebase JS SDK over HTTPS (Port 443).
- **CI-02:** File transfers for notes and attachments shall utilize Firebase Storage secure download URLs.
- **CI-03:** Real-time updates for attendance and notes shall be handled via Firestore WebSockets.

## 7. Use Case Descriptions

### 7.1 UC-01: Mark Attendance
- **Actor:** Student
- **Description:** The student enters a 6-digit code provided by the faculty to record their presence in a class session.
- **Pre-conditions:**
    1. Student is logged into the app.
    2. An attendance session is currently active for the student's group.
- **Post-conditions:**
    1. An attendance record is created in Firestore with a timestamp.
    2. The student's attendance history is updated.

### 7.2 UC-02: Download Course Material
- **Actor:** Student
- **Description:** The student selects a file attachment from the class stream and saves it to their device for offline viewing.
- **Pre-conditions:**
    1. Student has an active internet connection.
    2. The file exists in Firebase Storage.
- **Post-conditions:**
    1. The file is saved to the device's local filesystem.
    2. The student can open the file using a native document viewer.

### 7.3 UC-03: View Performance
- **Actor:** Student
- **Description:** The student checks their internal assessment marks and calculated average for a specific subject.
- **Pre-conditions:**
    1. Faculty has uploaded marks for the student.
- **Post-conditions:**
    1. The student sees a detailed breakdown of their scores and their current standing.
